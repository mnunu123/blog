## 쓰리봇 프로젝트 개요

![쓰리봇 웹앱 포트폴리오 — 프로젝트 전체 구조](/images/threebot-web/portfolio-p128.png)

쓰리봇(Threebot)은 **도시 빗물받이 자동 청소 로봇**이다. 물리적 로봇(하드웨어) + 원격 모니터링 웹앱(소프트웨어)으로 구성된다.

| 구성 | 기술 |
|------|------|
| **로봇 제어** | STM32 펌웨어, 모터 PWM 제어, ALM 센서 원점복귀 |
| **원격 조작** | 조이스틱 텔레오퍼레이션, 비상정지 로직 |
| **통신** | ROS2-임베디드 연동 |
| **웹 프론트엔드** | Next.js + 네이버 지도 API |
| **백엔드** | FastAPI + PostgreSQL (Docker) |

이 포스트는 **웹앱 파트** — 빗물받이 데이터를 지도 위에 시각화하는 관리 대시보드 개발 과정을 기록한다.

---

## 1. 지도 기반 빗물받이 시각화 시스템

**풀어야 했던 문제**: 수천 개의 빗물받이 데이터를 리스트로 보면 위험 구역을 파악하기 어렵다. 공간 정보를 활용하지 못하고 있었다.

![네이버 지도 연동 직후 — 빗물받이 마커가 지도 위에 올라간 첫 화면](/images/threebot/map-ui-naver-integration.png)

**접근 방법**: 네이버 지도에 마커 클러스터링을 도입하고, 상태별 색상으로 직관적인 구분을 추가했다.

```typescript
new naver.maps.Marker({
  position,
  icon: {
    content: `<div class="marker ${status}"></div>`
  }
})
```

상태 표시는 단순하게:
- 🔴 위험
- 🟡 주의
- 🟢 정상

**결과**: 관리자가 지도를 열면 위험 구역이 바로 눈에 들어온다. 데이터가 정보에서 의사결정 도구로 바뀐 순간이었다.

---

## 2. 백엔드 API 연동과 환경 변수 관리

**풀어야 했던 문제**: "데이터 없음" 오류가 자주 발생하고, 프론트↔백엔드 연결이 불안정했다. `.env` 설정이 제대로 반영되지 않은 문제였다.

**접근 방법**: 환경 변수 기반으로 API URL을 관리하고, 모든 API 호출 구조를 통일했다.

```bash
BACKEND_API_URL=http://localhost:8001
```

```typescript
fetch(`${process.env.BACKEND_API_URL}/drainage/markers`)
```

**결과**: 데이터 로딩이 안정화되고, 개발/배포 환경 분리가 가능해졌다.

---

## 3. 시군구 필터링 + 지도 이동 UX

**풀어야 했던 문제**: 필터에서 "대구광역시"를 선택해도 지도가 서울을 보고 있으면 의미가 없다. UX가 끊겼다.

**접근 방법**: 필터 선택 시 해당 지역의 중심 좌표로 지도를 이동시키도록 연결했다.

```typescript
map.setCenter(new naver.maps.LatLng(lat, lng))
map.setZoom(12)
```

**결과**: 필터 선택 = 즉시 공간 이동. 사용자 행동 흐름이 자연스러워졌다.

---

## 4. GeoJSON 행정구역 경계 시각화

**풀어야 했던 문제**: 데이터는 구 단위로 있는데 경계 표현이 없어서 어디서 어디까지가 관할인지 알 수 없었다.

**접근 방법**: GeoJSON 폴리곤으로 행정구역 경계를 렌더링했다.

```typescript
const polygon = new naver.maps.Polygon({
  paths,
  strokeColor: '#0EA5E9',
  fillColor: '#38BDF8',
  fillOpacity: 0.1
})
```

GeoJSON 속성 구조:
- `sido` — 시/도
- `sgg` — 시/군/구
- `adm` — 읍/면/동

**결과**: 행정구역 경계가 지도 위에 겹쳐지면서 데이터와 공간 구조가 결합됐다.

## 5. 시군구 토글 UI 3-state 버그

이번 개발에서 가장 인상 깊었던 디버깅 케이스다.

**증상**: 토글 버튼을 클릭하면 ON/OFF가 아니라 세 번 클릭해야 원래 상태로 돌아오는 현상.

```
1번째 클릭 → A 스타일
2번째 클릭 → B 스타일
3번째 클릭 → 다시 A  ← 이상하다
```

**원인 분석**: 폴리곤 레이어가 여러 개 생성되어 있고, `useEffect`가 각각 분리되어 독립적으로 동작하고 있었다. 각 effect가 서로 다른 폴리곤 인스턴스를 관리하면서 상태가 3가지처럼 보였다.

**해결**: 폴리곤 참조를 하나의 ref로 통합하고, 토글 시 항상 전체 초기화 → 재생성 방식으로 바꿨다.

```typescript
const polysRef = useRef<naver.maps.Polygon[]>([])

function clearPolys() {
  polysRef.current.forEach(p => p.setMap(null))
  polysRef.current = []
}

useEffect(() => {
  clearPolys()
  if (!show) return

  const polys = createPolygons()
  polys.forEach(p => p.setMap(map))
  polysRef.current = polys
}, [show])
```

**결과**: ON → 표시 / OFF → 완전 제거. 버그가 완전히 사라졌고 구조도 단순해졌다.

<Callout type="insight">
`useRef`로 외부 라이브러리 인스턴스(지도 폴리곤, 마커 등)를 관리할 때는 반드시 단일 ref로 통합하고 effect 시작 시 클리어부터 해야 한다. 분산된 ref는 상태 동기화 문제를 만든다.
</Callout>

## 6. 지도 시각 디자인 — 가독성과 투명도

![UI 리디자인 후 — LLM 사이드바와 지역 현황 패널이 추가된 화면](/images/threebot/map-ui-redesign.png)

**풀어야 했던 문제**: 경계 폴리곤 색상이 진해서 지도 자체가 안 보였다.

**접근 방법**: 행정구역 레벨별로 색상 계층을 설계하고, `fillOpacity`를 최소화했다.

| 레벨 | 색상 전략 |
|------|---------|
| 시/도 | 파란 계열 |
| 시/군/구 | 노란·연두 계열 |
| 읍/면/동 | 연한 색 |
| 선택 영역 | 강조 색 |

```typescript
fillOpacity: 0.14  // 핵심 개선 포인트
```

**결과**: 경계는 보이되 지도 정보를 가리지 않는 균형점을 찾았다.

---

## 7. 마커 클러스터 + 경계 레이어 충돌

**풀어야 했던 문제**: 마커와 폴리곤이 같은 영역에 겹치면서 시각 정보가 충돌했다.

**접근 방법**:
- `fillOpacity` 최소화로 폴리곤이 마커를 가리지 않게
- 클릭 이벤트를 레이어별로 분리해 인터랙션 충돌 방지
- 줌 레벨에 따라 레이어 가시성 분기

**결과**: 정보 계층 구조가 명확해지고 사용자 시선이 분산되지 않게 됐다.

## 8. 시스템 아키텍처 정리

```
[Frontend — Next.js]
       ↓ API 요청
[Backend — FastAPI]
       ↓
[DB / GeoJSON 파일]
```

- **프론트**: 시각화 & 인터랙션 (지도, 필터, 마커)
- **백엔드**: 데이터 처리 & API 제공 (FastAPI)
- **데이터**: 위치 기반 구조 (PostgreSQL + GeoJSON)

역할이 명확히 분리되면서 이후 AI 데이터, 로봇 센서 데이터를 붙이는 기반이 만들어졌다.
